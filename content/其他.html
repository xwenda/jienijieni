patch-package使用 ​

使用

1、安装npm i patch-package / yarn add patch-package

2、package.json的script中增加"postinstall": "patch-package"，此语句可在其他协作者安装依赖时，自动应用patch-package的修改

3、修改node_modules中需要修改的包并记下包名

4、npx patch-package 包名 => 生成patches文件夹，此为修改的内容

5、需要将patches包一并推入代码库中

(注意！patch-package需要用git，需要配置环境变量)

electron-update ​

electron自动更新

  1. 在根目录新建updater.js

    js
    const { autoUpdater } = require('electron-updater');
var mainWin = null;

const checkUpdate = (win, ipcMain) => {
  autoUpdater.autoDownload = true; // 自动下载
  autoUpdater.autoInstallOnAppQuit = true; // 应用退出后自动安装
  mainWin = win;
  // 检测是否有更新包并通知
  autoUpdater.checkForUpdatesAndNotify().catch();
  // 监听渲染进程的 install 事件，触发退出应用并安装
  ipcMain.handle('install', () => autoUpdater.quitAndInstall());

  autoUpdater.on('update-available', (info) => {
    console.log('有新版本需要更新');
  });
  autoUpdater.on('update-not-available', (info) => {
    console.log('无需更新');
  });
  autoUpdater.on('download-progress', (prog) => {
    mainWin.webContents.send('update', {
      speed: Math.ceil(prog.bytesPerSecond / 1000), // 网速
      percent: Math.ceil(prog.percent), // 百分比
    });
  });
  autoUpdater.on('update-downloaded', (info) => {
    mainWin.webContents.send('downloaded');
    // 下载完成后强制用户安装，不推荐
    autoUpdater.quitAndInstall();
  });
  
};

module.exports = checkUpdate;

  2. 在main.js引入

    js
    const { ..., ipcMain  } = require('electron')
const checkUpdate = require('./update.js');
...
const createWindow = () => {
  const win = ...
  ....
  checkUpdate(win, ipcMain);
  ...
}

  3. 在package.json的electron-builder配置中增加

    1. 其中url需要填一个可访问的服务器的静态文件夹地址
    2. version用于检测版本
    js
    {
  ...,
  "build": {
	...
    "publish": [
      {
        "provider": "generic",
        "url": "XXXX"
      }
    ]
  },
  ...,
  "version": "XXX"
}

  4. electron-builder打包，会生成latest.yml和安装包，

    1. 将这两个文件扔进服务器的静态文件夹中
    2. 尝试访问 XXXX/latest.yml（XXXX为服务器地址），若访问成功即成功了

关于接口延迟的测试方法 ​

如果出现一个接口超时 ​

网络问题： ​

如果仅仅只是网络延迟，所有接口都应该超时

接口串行问题： ​

该接口在后台等待了别的接口处理，这种情况的判断的方法：去接口拦截器把超时时间设置1到2分钟，看看实际请求时间，然后单独发这个请求，看看这个时间差

发现这个问题的场景 ​

在写路由鉴权时，没有把用户权限都请求回来，是不能放行的

js
router.beforeEach((to, from, next) => {
  NProgress.start()
  if (getToken()) {
    console.log('缓存存在TOKEN', getToken())
    store.dispatch('ValidToken', getToken()).then(() => {
      store.dispatch('GetUserRole', getToken())
      next()
    }).catch(err => {
      store.dispatch('Logout')
    })
  } else {
      ...
  } 
})

例如上图的写法，在没有进行 store.dispatch('GetUserRole', getToken())用户信息请求成功返回的情况下直接页面放行，此时会让页面所有接口发送

此时就会进入到上面所说的接口串行问题，导致获取用户信息的接口排在后面，等待了别的接口完成

正确的路由鉴权写法应该是

js
router.beforeEach((to, from, next) => {
  NProgress.start()
  if (getToken()) {
    console.log('缓存存在TOKEN', getToken())
    store.dispatch('ValidToken', getToken()).then(() => {
      store.dispatch('GetUserRole', getToken()).then(res => {
		next()
      })    
    }).catch(err => {
      store.dispatch('Logout')
    })
  } else {
      ...
  } 
})


 store.dispatch('GetUserRole', getToken()) 此处应该返回一个promise，包含的是最后的请求结果，即用户信息的返回，才能放行页面
